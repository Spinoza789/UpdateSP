#13 - FS3 Vendor Order Overhaul
What & Why
Rebuild the FS3 tab to be fully routing-aware. Orders are grouped by where parcels physically go (per reshipper / direct). Separate output sheets are generated per routing group. A pre-flight checklist prevents generating an incomplete or financially incorrect vendor order. Personal additions can be assigned to any routing group. A balance due review panel handles cost changes before submission. Live P&L updates as the batch is built. A vendor order history records every submission. GB order editing is locked when the batch is exported.
Done looks like
* FS3 opens to a batch builder grouped by routing destination: Direct Shipping section and one section per reshipper, each showing kit count, tier, total vendor shipping cost, and per-order share sourced from the database
* Orders with unconfirmed payment are excluded by default; including them requires a manual override that is logged
* A persistent warning panel shows: orders with missing wholesale prices (with inline "Add Price" action), orders with unconfirmed payment included via override, orders amended after batch lock, price changes since the GB opened
* Filters: routing type, payment status, country, individual order toggle (include/exclude per order with reason)
* Excluded orders are shown in a separate panel with their exclusion reason; they can be reinstated before generation
* Personal additions can be assigned to any routing group; doing so contributes to that group's kit count and tier calculation with a cost impact preview before confirming
* Admin can choose to absorb or spread personal addition cost across the batch
* A pre-flight checklist must be fully green before the export button is active: all included products have wholesale prices, all direct orders have complete addresses, all included orders have confirmed payment or explicit override, all balance dues are resolved
* Balance due review panel shows each affected order with old amount, new amount, and difference — actions: absorb, request balance (customer not notified until admin triggers), hold order (moves to excluded)
* Currency is USD throughout unless the GB has a currency override specified in GB settings
* "Generate Vendor Order" produces separate downloadable sheets: one per reshipper (reshipper address, product totals, kit count, shipping tier, total cost) and one for direct (each order individually with customer name, address, products, quantities, individual shipping cost)
* A combined summary sheet (all routing groups merged) is available as an optional additional download
* Marking the batch as submitted locks all included orders (batch_locked = true), sets their status to Processing, records submission timestamp, and writes to routing_history
* Once locked, routing changes to those orders require an admin override with logged reason
* After lock, the FS3 "re-open" action unlocks the batch with a mandatory reason, notifies via audit log, and restores routing editability
* Vendor order history tab records every submission: date, GB, total orders, total kits, routing breakdown, who generated it, download links to regenerate the original sheets
* Live P&L panel updates as orders are added/removed: product subtotal, wholesale cost, vendor shipping, reshipper fees, delivery collected, tips, admin fee — gross margin per product line with traffic light indicator (green >30%, amber 10–30%, red <10%)
* Per-reshipper P&L shown separately alongside overall GB P&L
* Personal additions P&L tracked separately from group buy orders
* Wholesale price change alerts shown as a banner if any price in "My Prices" has changed since the GB opened, with affected products and P&L impact shown
* All errors during sheet generation are logged to audit_logs and shown as inline error messages — generation does not silently fail
Out of scope
* Changes to the "My Prices" CRUD editor (existing functionality preserved as-is)
* Changes to the import prices CSV tool (existing functionality preserved as-is)
* Changes to the Ping Missing Address tool (existing functionality preserved as-is, but missing addresses also flagged in the warning panel)
* Customer notification triggering (admin manually sends balance due notifications from the order page, not from FS3)
Steps
1. Build the routing-aware batch builder view — Replace the current single combined order list with a grouped view: one collapsible section per reshipper (showing their name, hub country flag, kit count, tier, total vendor shipping cost, per-order share) and one Direct Shipping section. Each section lists its orders with payment status, kit count, and include/exclude toggle. Data is loaded from GET /admin/group-buys/:gbId/fs3-batch which returns orders pre-grouped by routing_type and reshipper_username.
2. Build the excluded orders panel — Orders excluded from the batch appear in a collapsible panel below the main groups with their exclusion reason (unconfirmed payment, missing address, missing price, manually excluded). Each excluded order has a "Reinstate" button that moves it back into its routing group. Exclusion reasons are stored in component state only (not persisted) since they only matter during the current session of building the batch.
3. Build the warning panel for FS3 — Persistent panel at the top of the FS3 view sourced from GET /admin/group-buys/:gbId/routing-warnings plus a new GET /admin/group-buys/:gbId/fs3-warnings endpoint that adds: missing wholesale prices (with affected product names and inline "Add Price" input that saves to the existing fs3-costs endpoint), wholesale price changes since GB opened (with old price, new price, and P&L impact per product).
4. Build personal additions routing assignment — Extend the existing Personal Additions feature with a routing group selector. When an addition is assigned to a reshipper group, it contributes to that group's kit count. Show a cost impact preview (same mechanism as the board) before confirming the addition. Admin can choose to absorb the additional vendor shipping cost or spread it across confirmed orders in the batch. Personal additions appear as a separate line in the output sheet for their routing group, clearly labelled.
5. Build the balance due review panel — Before the pre-flight checklist runs, surface orders where the vendor shipping has changed since the customer last paid. For each: show order code, username, old amount paid, new amount owed, difference. Actions: "Absorb" (sets amount_due = 0 for the difference, logs to routing_history), "Request Balance" (sets amount_due, marks order for customer notification — notification is not sent until admin triggers it separately from the order page), "Hold" (moves order to excluded panel). Panel must be empty (all decisions made) before pre-flight proceeds.
6. Build the pre-flight checklist — A checklist component that evaluates before enabling the export button. Each item is a pass/fail check with a count of affected orders and a link to fix them: (1) All included products have wholesale prices set, (2) All direct orders have complete shipping addresses, (3) All included orders have confirmed payment or manual override, (4) All balance dues resolved, (5) No orders amended after a previous batch lock without review. The export button is disabled (greyed, not hidden) with a tooltip explaining which checks are failing.
7. Build the vendor order sheet generator — A new backend endpoint POST /admin/group-buys/:gbId/fs3-generate accepts the finalised batch (list of order IDs grouped by routing destination) and returns: one text file per reshipper containing their address, product totals, kit count, tier label, and total shipping cost; one text file for direct orders with each order as an individual block (customer name, full address, products and quantities, individual shipping cost); one optional combined summary. Files are generated server-side and returned as downloadable blobs. Personal additions are included in the relevant group's sheet as a clearly labelled separate section.
8. Implement batch submission and locking — POST /admin/group-buys/:gbId/fs3-submit marks all included orders with batch_locked = true, batch_locked_at = now(), status = processing, and writes a routing_history entry with changeType = batch_locked for each order. Returns confirmation with order count and submission timestamp. The FS3 view switches to a "Submitted" state showing the locked batch details and a "Re-open Batch" action.
9. Build vendor order history tab — A new tab within FS3 listing all past submissions for the GB: date, total orders, total kits per routing group, who generated it, and a "Regenerate Sheets" button that re-runs the generator with the original batch data (from the stored submission record). History records are stored in a new fs3_submissions table with gbId, submittedBy, submittedAt, batchSnapshot (jsonb of the order IDs and routing at time of submission).
10. Build the live P&L panel — A sidebar panel that recalculates on every include/exclude toggle and personal addition change. Shows: product subtotal (customer prices), wholesale cost (from My Prices), vendor shipping (from dynamic calculator), reshipper fees, delivery collected, tips, admin fees, gross margin, and effective margin percentage. Per-product breakdown with traffic light margin indicators. Per-reshipper sub-totals alongside the overall GB total. Personal additions shown as a separate P&L line. Currency displayed per GB settings (default USD).
11. Add FS3 filters and order search — Filter bar above the batch builder: routing type selector, payment status filter, country filter, text search by username or order code. Filters are applied server-side via query params on the fs3-batch endpoint. Individual order cards within each group have an include/exclude toggle and an expand action to see full order details without leaving FS3.
12. Add price change alert banner — On load, GET /admin/group-buys/:gbId/fs3-price-changes compares current "My Prices" values against prices at the time of the first order in the GB (stored as a snapshot when the GB opens). Returns a list of changed products with old price, new price, and calculated P&L impact across affected orders. Shown as a dismissible banner. Dismissal is per-session only — the check re-runs on next FS3 load.
