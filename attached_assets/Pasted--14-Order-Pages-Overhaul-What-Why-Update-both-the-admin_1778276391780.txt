#14 - Order Pages Overhaul
What & Why
Update both the admin orders page and the customer order portal to reflect the new routing model. Admin orders page moves to server-side filtering with pagination, gains routing-aware display, batch lock enforcement, routing history timeline, and auto-saving line item drafts. Customer portal removes the self-service direct shipping toggle, adds reshipper code entry at checkout, and shows clear routing status and balance due breakdowns. All changes protect existing paid orders — no financial data changes unless admin explicitly overrides.
Done looks like
* Admin orders page filters are applied server-side; the browser never loads all orders into memory simultaneously
* Pagination is in place — admin pages through results rather than scrolling an infinite list
* Each order card routing tab shows: routing_type badge (Direct / Reshipper / Unrouted / Legacy), assigned reshipper with hub country flag, reshipper code used for assignment (if applicable), cross-border indicator when customer country ≠ reshipper hub, batch lock badge (with lock date), routing history timeline (every change with who made it and when)
* Changing routing type or reshipper from the orders page calls the same routing mutation endpoint as the board — changes immediately reflect on the fulfillment board
* Batch-locked orders show all routing fields as read-only with a clear "Locked — in production with vendor" label; edit requires an override with reason
* Line item edits auto-save to the database as a draft every 30 seconds; a "draft saved" indicator confirms this; navigating away does not lose changes
* When a confirmed-payment order has a changed vendor shipping amount, a "Balance Due Review" badge appears on the order card with the difference shown; admin resolves it from the card (absorb or request) before the badge clears
* New filter options on the admin orders page: routing type (Direct / Reshipper / Unrouted / Legacy), batch lock status (Locked / Unlocked), balance due review needed, reshipper code used
* Warning indicators on the orders list: orders with missing addresses show a warning icon; orders with unresolved balance dues show a flag; batch-locked orders show a lock icon — all visible without expanding the card
* Customer portal removes the direct shipping self-service toggle
* Customer portal has reshipper code entry field at the order placement step (OrderForm) — if a code is entered and valid, routing is assigned immediately; if no code, only direct shipping is available for GBs with coded reshippers
* Customer portal shows routing status clearly once confirmed: "Your order ships directly to your address" or "Your order is being handled by [Reshipper Name]" — shown only after admin confirms routing, not before
* Customer portal shows "Routing being finalised" when routing_type is unrouted or null
* Customer portal shows vendor shipping as the calculated amount (not TBD) once routing is confirmed and the batch calculation has run; shows TBD only while routing is genuinely unresolved
* Customer balance due notification shows a clear breakdown: original amount paid, new amount owed, difference, and reason ("your shipping batch was updated") — not a raw number
* Customer address editing is disabled when batch_locked = true with a message "Your order is in production — contact us to update your address"
* All changes write to the database immediately; no routing or financial state lives in the browser
Out of scope
* Reshipper-facing portal (separate feature, not in this task)
* FS3 changes (Task 3)
* Fulfillment board changes (Task 2)
* Removing legacy fields from the database
Steps
1. Move admin orders filtering to the server — Refactor GET /admin/orders to accept filter query parameters: status, paymentStatus, paymentMethod, gbId, reshipperUsername, country, routingType, batchLocked, balanceDueReview, dateFrom, dateTo, search. Return paginated results (default 50 per page) with total count. Remove the large client-side useMemo filter block in Admin.tsx and replace with a query that sends filter state to the API. Preserve all existing filter options — add the new routing-specific ones alongside.
2. Add pagination to the admin orders list — Add page controls (previous/next, page number, results per page selector) to the orders tab. Current page and filter state are stored in the URL as query params so they survive refresh. Loading state shows a skeleton while the new page fetches — the list does not go blank.
3. Update the order card routing tab — Replace the current direct shipping toggle and force-assign reshipper dropdown with: routing_type selector (Direct / Reshipper / Unrouted), reshipper assignment dropdown (appears when type is Reshipper), country leg correction dropdown, cross-border indicator, reshipper code display, batch lock status with lock date. Routing type changes call PATCH /admin/orders/:id/routing. Batch-locked orders render all fields as read-only with an "Override Lock" button that requires a reason before unlocking.
4. Add routing history timeline to order card — A collapsible "History" section within the routing tab. Fetches from GET /admin/orders/:id/routing-history on expand. Renders a chronological timeline: timestamp, changed by, change type (routing_assigned, reshipper_changed, cost_recalculated, financial_override, batch_locked, error), previous value, new value, reason (for overrides). Errors in the history are shown with a red indicator.
5. Add warning icons to order list items — Without expanding a card, show inline badges for: missing address (orange warning icon), unresolved balance due (amber flag), batch locked (grey lock icon), balance due review needed (red flag). These are included in the paginated API response as boolean flags so they render without an additional fetch.
6. Add new filter options to the admin orders filter bar — Extend the existing filter UI with: routing type multi-select (Direct, Reshipper, Unrouted, Legacy), batch lock status toggle (Locked / Unlocked), balance due review toggle (needs review / all), reshipper code filter (text input matching the code used for assignment). All filters send to the server — no client-side filtering.
7. Implement auto-saving line item drafts — When an admin opens the line items editor on an order, the current state is written to a draft_line_items jsonb field on the order every 30 seconds via PATCH /admin/orders/:id/line-items-draft. A "Draft saved [time]" indicator confirms the save. When the editor is opened on an order that has a draft, a banner asks "You have unsaved changes from [time] — restore or discard?" Committing line items clears the draft. This replaces the current browser-only draft state.
8. Build balance due resolution on order card — When amount_due > 0 and payment_status = confirmed, a "Balance Due Review" section appears in the order card showing: original confirmed amount, current total, difference, and two actions: "Absorb" (sets amount_due = 0, logs reason to routing_history, no customer notification) and "Request Balance" (keeps amount_due, marks order for customer notification — notification is sent separately via the existing Telegram ping mechanism, not automatically). Badge clears once a decision is made.
9. Remove customer-facing direct shipping toggle from OrderForm — In OrderForm.tsx, remove the UI element that allows customers to self-select direct shipping. Replace with a reshipper code input field: a text input labelled "Reshipper Code (if provided by your reshipper)" with a validate-on-blur call to GET /api/group-buys/:gbId/validate-reshipper-code?code=X which returns the reshipper's display name (not their username) if valid, or an error if invalid or at capacity. If a valid code is entered, the order is flagged for that reshipper assignment on submission. If no code is entered and the leg has coded reshippers, only direct shipping is presented. If no code is entered and the leg has no coded reshippers, existing assignment logic applies.
10. Update customer order portal routing display — In AccountOrderDetail.tsx, replace the direct shipping toggle display with a routing status component. States: "Routing being finalised" (routing_type = unrouted or NULL), "Your order ships directly to your address" (routing_type = direct), "Your order is being handled by [Reshipper Display Name]" (routing_type = reshipper — show display name only, not username). Routing status only shows once confirmed — the unresolved state shows the neutral message.
11. Update vendor shipping display on customer portal — Replace the "TBD" placeholder with a state machine: "To be calculated" (routing not yet confirmed), calculated amount (routing confirmed and batch calculation run), "Updated — see balance due" (amount changed since last payment). When showing "Updated", display a breakdown: "Previously: $X | Updated: $Y | Difference: $Z — Your shipping batch was updated."
12. Enforce batch lock on customer portal — When batch_locked = true, disable the address editing field with message: "Your order is currently in production with our supplier. To update your address, please contact us directly." Hide the direct shipping cost recalculation button if it exists. Show a status badge "In Production" alongside the existing order status.
13. Add reshipper code validation endpoint — GET /api/group-buys/:gbId/validate-reshipper-code?code=X — public endpoint (no auth required beyond GB membership). Returns: { valid: true, reshipperDisplayName: "EU Reshipper", remainingCapacity: null } or { valid: false, reason: "invalid_code" | "inactive" | "at_capacity" }. When at_capacity, the frontend shows "This reshipper is currently full — please contact your group buy organiser" and falls back to direct shipping only.
